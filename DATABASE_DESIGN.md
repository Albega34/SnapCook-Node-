# 🗄️ SnapAndCook — Database Design Documentation

> **Database Engine:** MongoDB (NoSQL, Document-Oriented)  
> **ODM:** Mongoose v7+  
> **Hosting:** MongoDB Atlas (Cloud)  
> **Connection:** Via `MONGO_URI` in `.env`

---

## 📐 Overview

SnapAndCook uses **MongoDB** as its primary database. All data is stored as JSON-like documents across **4 collections**. The schema is defined and enforced using **Mongoose** models in the `backend/models/` directory.

```
MongoDB Atlas
│
├── recipes          ← All AI-scanned food recipes
├── users            ← User profile & dietary preferences
├── shoppinglists    ← Per-user ingredient shopping list
└── mealplans        ← Weekly meal planner entries
```

---

## 📦 Collection 1: `recipes`

> **Model File:** `backend/models/Recipe.js`  
> **Mongoose Model:** `Recipe`  
> **Purpose:** Stores every food recipe that was scanned via the AI pipeline. Each document is auto-generated by the Gemini Vision API and saved after a `/api/scan` call.

---

### Schema Definition

```js
const recipeSchema = new mongoose.Schema({
  id:          { type: String, required: true, unique: true },
  title:       String,
  description: String,
  image:       String,
  confidence:  Number,
  cuisine:     String,
  calories:    Number,
  protein:     Number,
  fats:        Number,
  carbs:       Number,
  time:        Number,
  rating:      Number,
  reviews:     String,
  difficulty:  String,
  tags:        [String],
  servings:    Number,
  ingredients: [ingredientSchema],
  instructions:[instructionSchema],
  history:     historySchema,
  trends:      [trendSchema]
}, { timestamps: true });
```

---

### Field Reference

| Field         | Type     | Required | Unique | Default | Description                                         |
|---------------|----------|----------|--------|---------|-----------------------------------------------------|
| `_id`         | ObjectId | Auto     | ✅     | Auto    | MongoDB auto-generated primary key                  |
| `id`          | String   | ✅       | ✅     | —       | Custom ID like `scanned-1714900000000`              |
| `title`       | String   | ❌       | ❌     | —       | Name of the dish (e.g., "Butter Chicken")           |
| `description` | String   | ❌       | ❌     | —       | Short appetizing description of the dish            |
| `image`       | String   | ❌       | ❌     | —       | ImgBB-hosted image URL                              |
| `confidence`  | Number   | ❌       | ❌     | —       | AI confidence score (0–100)                         |
| `cuisine`     | String   | ❌       | ❌     | —       | Cuisine type (e.g., "Indian", "Italian")            |
| `calories`    | Number   | ❌       | ❌     | —       | Estimated calories per serving                      |
| `protein`     | Number   | ❌       | ❌     | —       | Protein content in grams                            |
| `fats`        | Number   | ❌       | ❌     | —       | Fat content in grams                                |
| `carbs`       | Number   | ❌       | ❌     | —       | Carbohydrate content in grams                       |
| `time`        | Number   | ❌       | ❌     | —       | Total cooking/prep time in minutes                  |
| `rating`      | Number   | ❌       | ❌     | —       | Quality rating (1.0 – 5.0)                          |
| `reviews`     | String   | ❌       | ❌     | —       | Review count label (e.g., "120 Reviews")            |
| `difficulty`  | String   | ❌       | ❌     | —       | `"Easy"`, `"Medium"`, or `"Hard"`                   |
| `tags`        | [String] | ❌       | ❌     | `[]`    | Category labels (e.g., `["Vegan", "Curry"]`)        |
| `servings`    | Number   | ❌       | ❌     | —       | Number of servings the recipe yields                |
| `ingredients` | [Object] | ❌       | ❌     | `[]`    | Array of ingredient sub-documents (see below)       |
| `instructions`| [Object] | ❌       | ❌     | `[]`    | Array of step-by-step instruction sub-documents     |
| `history`     | Object   | ❌       | ❌     | —       | Cultural/historical background of the dish          |
| `trends`      | [Object] | ❌       | ❌     | `[]`    | Country-wise consumption popularity data            |
| `createdAt`   | Date     | Auto     | ❌     | Auto    | Mongoose timestamp — when the doc was created       |
| `updatedAt`   | Date     | Auto     | ❌     | Auto    | Mongoose timestamp — last updated time              |

### Sub-Schema: `ingredients[]`

Each ingredient is an embedded sub-document:

```js
const ingredientSchema = new mongoose.Schema({
  name:        String,
  checked:     { type: Boolean, default: false },
  substitutes: [String]
});
```

| Field         | Type     | Default | Description                                            |
|---------------|----------|---------|--------------------------------------------------------|
| `name`        | String   | —       | Ingredient name (e.g., "Chicken Breast")               |
| `checked`     | Boolean  | `false` | Whether the user has checked off this ingredient       |
| `substitutes` | [String] | `[]`    | Healthier/dietary alternatives (1–2 options)           |

**Example:**
```json
{
  "name": "Heavy Cream",
  "checked": false,
  "substitutes": ["Coconut Cream", "Cashew Milk"]
}
```

### Sub-Schema: `instructions[]`

Each step in the cooking process:

```js
const instructionSchema = new mongoose.Schema({
  step:  Number,
  title: String,
  time:  String,
  desc:  String
});
```

| Field   | Type   | Description                                     |
|---------|--------|-------------------------------------------------|
| `step`  | Number | Step number (1, 2, 3 ...)                        |
| `title` | String | Short step heading (e.g., "Marinate the Chicken")|
| `time`  | String | Estimated time for this step (e.g., `"15m"`)    |
| `desc`  | String | Full description of what to do in this step      |

**Example:**
```json
{
  "step": 1,
  "title": "Marinate the Chicken",
  "time": "20m",
  "desc": "Mix yogurt, turmeric, chili powder, and garam masala. Coat chicken pieces and let it rest for 20 minutes."
}
```

### Sub-Schema: `history`

Cultural background of the dish:

```js
const historySchema = new mongoose.Schema({
  story: String,
  image: String,
  rank:  String
});
```

| Field   | Type   | Description                                         |
|---------|--------|-----------------------------------------------------|
| `story` | String | Fascinating cultural or historical origin story     |
| `image` | String | Image URL representing the dish or its origin       |
| `rank`  | String | Global popularity ranking (e.g., `"#4 Global"`)    |

**Example:**
```json

  "image": "https://i.ibb.co/xyz.jpg",

```

## 👤 Collection 2: `users`

> **Model File:** `backend/models/User.js`  
> **Mongoose Model:** `User`  
> **Purpose:** Stores the user profile including name and dietary preferences. Currently uses a single default user (`id: "default_user"`). The profile is auto-created on first access.

---

### Schema Definition

```js
const userSchema = new mongoose.Schema({
  id:   { type: String, required: true, unique: true },
  name: String,
  dietaryPreferences: {
    vegan:       { type: Boolean, default: false },
    vegetarian:  { type: Boolean, default: false },
    keto:        { type: Boolean, default: false },
    glutenFree:  { type: Boolean, default: false }
  },
  dailyCaloriesGoal: { type: Number, default: 2000 }
}, { timestamps: true });
```

---

### Field Reference

| Field                        | Type    | Required | Unique | Default        | Description                                   |
|------------------------------|---------|----------|--------|----------------|-----------------------------------------------|
| `_id`                        | ObjectId| Auto     | ✅     | Auto           | MongoDB auto-generated primary key            |
| `id`                         | String  | ✅       | ✅     | —              | Custom user identifier (`"default_user"`)     |
| `name`                       | String  | ❌       | ❌     | —              | Display name (e.g., `"Chef"`)                 |
| `dietaryPreferences.vegan`   | Boolean | ❌       | ❌     | `false`        | Is the user vegan?                            |
| `dietaryPreferences.vegetarian`| Boolean| ❌      | ❌     | `false`        | Is the user vegetarian?                       |
| `dietaryPreferences.keto`    | Boolean | ❌       | ❌     | `false`        | Is the user on a keto diet?                   |
| `dietaryPreferences.glutenFree`| Boolean| ❌      | ❌     | `false`        | Is the user gluten-free?                      |
| `dailyCaloriesGoal`          | Number  | ❌       | ❌     | `2000`         | Target daily calorie intake                   |
| `createdAt`                  | Date    | Auto     | ❌     | Auto           | Mongoose timestamp                            |
| `updatedAt`                  | Date    | Auto     | ❌     | Auto           | Mongoose timestamp                            |

## 🛒 Collection 3: `shoppinglists`

> **Model File:** `backend/models/ShoppingList.js`  
> **Mongoose Model:** `ShoppingList`  
> **Purpose:** Stores the user's ingredient shopping list. Items are embedded sub-documents. Supports adding ingredients from recipes and removing them individually.

---

### Schema Definition

```js
const shoppingListItemSchema = new mongoose.Schema({
  name:    { type: String, required: true },
  checked: { type: Boolean, default: false }
});

const shoppingListSchema = new mongoose.Schema({
  userId: { type: String, default: "default_user" },
  items:  [shoppingListItemSchema]
}, { timestamps: true });
```

---

### Field Reference

| Field          | Type     | Required | Default          | Description                                      |
|----------------|----------|----------|------------------|--------------------------------------------------|
| `_id`          | ObjectId | Auto     | Auto             | MongoDB auto-generated primary key               |
| `userId`       | String   | ❌       | `"default_user"` | References the owning user                       |
| `items`        | [Object] | ❌       | `[]`             | Array of shopping item sub-documents             |
| `items[].name` | String   | ✅       | —                | Ingredient name (e.g., `"Garlic"`)               |
| `items[].checked`| Boolean| ❌       | `false`          | Whether the user has bought/checked off the item |
| `createdAt`    | Date     | Auto     | Auto             | Mongoose timestamp                               |
| `updatedAt`    | Date     | Auto     | Auto             | Mongoose timestamp                               |

---

### Sample ShoppingList Document

```json
{
  "_id": "664a1b2f9c3d001e7f5e4a03",
  "userId": "default_user",
  "items": [
    { "_id": "664a1c...", "name": "Chicken Breast", "checked": false },
    { "_id": "664a1d...", "name": "Garlic",          "checked": true  },
    { "_id": "664a1e...", "name": "Tomato Puree",    "checked": false },
    { "_id": "664a1f...", "name": "Heavy Cream",     "checked": false }
  ],
  "createdAt": "2024-05-03T10:00:00.000Z",
  "updatedAt": "2024-05-05T07:45:00.000Z"
}
```

> **Note:** Duplicate detection is enforced at the **API level** (case-insensitive), not at the DB schema level. If "Garlic" already exists, a second "garlic" won't be inserted.

## 📅 Collection 4: `mealplans`

> **Model File:** `backend/models/MealPlan.js`  
> **Mongoose Model:** `MealPlan`  
> **Purpose:** Stores weekly meal planning entries. Each document maps one meal slot (a specific day + meal type) to a recipe. Multiple documents together represent the full weekly plan.

---

### Schema Definition

```js
const mealPlanSchema = new mongoose.Schema({
  userId:      { type: String, default: "default_user" },
  day:         { type: String, required: true },   // "Monday", "Tuesday", ...
  meal:        { type: String, required: true },   // "Breakfast", "Lunch", "Dinner"
  recipeTitle: { type: String, required: true },
  recipeId:    { type: String }
}, { timestamps: true });
```

---

### Field Reference

| Field         | Type     | Required | Default          | Description                                         |
|---------------|----------|----------|------------------|-----------------------------------------------------|
| `_id`         | ObjectId | Auto     | Auto             | MongoDB auto-generated primary key                  |
| `userId`      | String   | ❌       | `"default_user"` | References the owning user                          |
| `day`         | String   | ✅       | —                | Day of week: `"Monday"` – `"Sunday"`                |
| `meal`        | String   | ✅       | —                | Meal slot: `"Breakfast"`, `"Lunch"`, or `"Dinner"`  |
| `recipeTitle` | String   | ✅       | —                | Display name of the planned recipe                  |
| `recipeId`    | String   | ❌       | —                | Custom `id` from the `recipes` collection (soft ref)|
| `createdAt`   | Date     | Auto     | Auto             | Mongoose timestamp                                  |
| `updatedAt`   | Date     | Auto     | Auto             | Mongoose timestamp                                  |

## 🔗 Collection Relationships

SnapAndCook uses a **soft-reference pattern** (no enforced foreign keys, as is typical in MongoDB). Relationships are maintained via matching `id` / `userId` string fields.

```
┌─────────────────────────────────────────────────────────────────────┐
│                        COLLECTIONS                                  │
│                                                                     │
│  ┌───────────┐         ┌──────────────────────────────────────┐    │
│  │  users    │         │            recipes                   │    │
│  │───────────│         │──────────────────────────────────────│    │
│  │ id ◄──────┼─────────┼── (soft ref via userId in other     │    │
│  │ name      │         │    collections)                      │    │
│  │ diet prefs│         │ id (custom string)                   │    │
│  │ calGoal   │         │ title, cuisine, nutrition...         │    │
│  └───────────┘         │ ingredients[], instructions[]        │    │
│        │               │ history{}, trends[]                  │    │
│        │               └──────────────────────────────────────┘    │
│        │                           ▲                               │
│        │                           │ recipeId (soft ref)           │
│        ▼                           │                               │
│  ┌──────────────┐     ┌────────────┴──────────┐                   │
│  │ shoppinglists│     │       mealplans        │                   │
│  │──────────────│     │────────────────────────│                   │
│  │ userId       │     │ userId                 │                   │
│  │ items[]      │     │ day                    │                   │
│  │  ├ name      │     │ meal                   │                   │
│  │  └ checked   │     │ recipeTitle            │                   │
│  └──────────────┘     │ recipeId               │                   │
│                        └────────────────────────┘                  │
└─────────────────────────────────────────────────────────────────────┘
```

### Relationship Summary

| From Collection  | References       | Via Field           | Type                    |
|------------------|------------------|---------------------|-------------------------|
| `shoppinglists`  | `users`          | `userId`            | Soft reference (String) |
| `mealplans`      | `users`          | `userId`            | Soft reference (String) |
| `mealplans`      | `recipes`        | `recipeId`          | Soft reference (String) |

> **Why soft references?** MongoDB doesn't enforce foreign key constraints. The application handles consistency at the API layer. This keeps the schema flexible and avoids expensive joins.

## 📏 Indexes

| Collection      | Field   | Index Type | Reason                                        |
|-----------------|---------|------------|-----------------------------------------------|
| `recipes`       | `id`    | Unique     | Custom ID lookups (`GET /api/recipes/:id`)    |
| `users`         | `id`    | Unique     | Single-user lookup by custom string ID        |
| `shoppinglists` | `userId`| Normal     | Filter lists by user (implicit via `findOne`) |
| `mealplans`     | `userId`| Normal     | Fetch all plans for a user                    |

> All `unique: true` fields in Mongoose automatically create a MongoDB unique index.

## 📊 Data Volume Estimates

| Collection      | Expected Growth         | Notes                                     |
|-----------------|------------------------|-------------------------------------------|
| `recipes`       | ~1 doc per scan        | Grows each time user scans a food image   |
| `users`         | Static (1 user)        | Single `default_user` document            |
| `shoppinglists` | 1 doc, many sub-items  | Items added/removed frequently            |
| `mealplans`     | Max 21 docs (7×3 slots)| One per day+meal combination              |

## ⚙️ Mongoose Configuration

All schemas use:
```js
{ timestamps: true }
```
This auto-adds `createdAt` and `updatedAt` fields to every document.

**Connection Setup** (`backend/config/db.js`):
```js
import mongoose from "mongoose";

export const connectDB = async () => {
  await mongoose.connect(process.env.MONGO_URI);
  console.log("MongoDB Connected");
};
```
